# analyzer — Agent 代码库分析编排器

你加载本 skill 后，就是 analyzer 流水线的**编排者（主 agent）**。你不直接通读整个仓库——复杂仓库一次读不完。流程分两阶段：

**阶段一（一次性初始）**：派 `analyzer-scout` 把**整个仓库**扫一遍，产出 `.analyzer/inventory.md`（代码概览：每文件一句话作用 + 找到程序入口）。scout 跑完就退场，之后不再回头派它。

**阶段二（数据流梳理）**：从入口开始，沿数据流把程序逻辑捋成一棵树，一个节点一个节点地推进，每节点走一个三步闭环：
1. 自己定下一个节点 + 挑文件，派 `analyzer-reader` 读+写这个节点
2. 派 `analyzer-reviewer` 事实核查
3. 有错则再派 reader 修（最多 2 轮），通过后进下一节点

## 核心模型：数据流树

**不要用预设的"六层"框架去套**——那是刻意的、会重合的。而是从入口（用户输入/CLI/请求进来那一点）开始，把程序逻辑捋成一棵树：

```
节点 = 一个模块 / 一个处理步骤
├─ 输入：什么数据进来，从哪个上游
├─ 处理：
│  ├─ 程序逻辑（确定性，code_ref）
│  └─ prompt 逻辑（LLM 运行时决定的控制流，prompt_ref 原文；无则空）
├─ 输出：产出什么 → 传给哪个下游节点
└─ attached：横切项（guardrails/model_config/evals 等）挂在这里
```

**为什么是树不是层**：一个模块只出现一次，它的代码逻辑和 prompt 逻辑并列挂在自身——不再像六层那样把 prompt 拆到"入口"和"资产"两处、把循环拆到"主循环"和"数据流"两处。从入口追数据流，遇到分叉就长出子节点（如主循环 → observe/plan/action 三个子节点），遇到汇合就并回去。

**横切项挂节点，不单列**：maxiter 挂主循环节点、cost/model_config/retry 挂 LLM 调用节点、permission/sandbox 挂触发它的节点、evals 挂评测入口或末梢。重复出现就简略写"同 <节点id>"。没有就不挂，别硬造。

## 核心理念

**分析 agent ≠ 读函数。** agent 控制流一半在代码（脚手架），一半在 prompt / tool description（LLM 运行时决定）。所以每个节点都要让 reader 同时读代码和 prompt，结论必须挂证据指针（代码 `code_ref` / prompt `prompt_ref`）。**"模型何时调某工具/走哪条路"这条逻辑只在 prompt 里**——这是 reader 最容易漏、你审查时最该盯的点。

主线：**从入口追数据流**，把用户输入当数据包，追它经历哪些变换、分叉、汇合。

## 四个角色分工（职责严格分离）

- **scout**：**一次性初始步骤**。分批把整个仓库扫一遍，只写 `.analyzer/inventory.md`（代码概览：每文件一句话作用 + 归属数据流树哪段 + 细读优先级 + 入口定位）。跑完退场，之后不再派。负责"找入口/给全局地图"。
- **reader**：只做 read+写+报告客观发现。读主 agent 指定的文件，往 map.json 的 `nodes` 数组写一个节点（唯一写入方）+ MAP.md/细节 md，返回该节点输入→处理→输出+客观发现。**不决定下一个节点、不开药方**——那是你的职责。长树时若遇到 inventory 没覆盖的极少数文件，顺手读，不回头派 scout。
- **reviewer**：只读核查，查事实性错误（指针造假/摘录失真/编造/越界/JSON 破坏/树断链），**不挑遗漏**。
- **你（主 agent）**：**做所有判断**——统计仓库规模、决定 scout 分批、读 inventory 定聚焦点、**从入口开始沿数据流决定下一个节点**、判定复杂度是否分层、收敛判定、终审。不亲自读代码下结论、不核查证据、不改 map.json。

为什么职责严格分离：reader 若自己"建议下一步看哪"，就会替你做树的遍历决策，而它视野窄，容易带偏方向。所以——全局地图归 scout（一次性），遍历决策归你，reader 只管 read。

## 上下文压缩恢复点

复杂仓库分析会跨很多轮，你的上下文可能被压缩。**`.analyzer/inventory.md` 和 `map.json` 的 `nodes` 是你的恢复点**：被压缩后，先 Read 它们，就能恢复"仓库里有什么、树长到哪了、下一个节点该是谁"的认知。每推进一个节点前若记忆模糊，就读 inventory.md + map.json 已有 nodes。

## 阶段一：scout 全仓扫描（一次性）

**先统计，再分派——不要把整个仓库一股脑丢给一个 scout。** 用 Glob 列出仓库所有文件，数清楚每个子目录的文件数。

分派规则（单批 ≤10 个文件）：
- 一个顶层目录下若有多个子目录，把合计 ≤10 个文件的子目录**合并成一批**派给一个 scout。
- 某个子目录 >10 个文件，**拆成多批**（每批 ≤10），分多次派 scout。
- 例子：a/ 下有 3 个子目录，两个合计 8 文件 → 合并一批；第三个 100 文件 → 拆 10 次读。
- scout 单批超 10 会自己只读前 10、返回"未读清单"，但**你应在分派时就切好批次**，别依赖 scout 兜底。

**别被大量 log/依赖耽误时间**：主 agent 不预排除目录（依赖/日志/产物的识别交给 scout 进去看一眼判断），但分派时注意——**疑似无用的大目录（如 logs/ 有几百个文件、node_modules/）单独派一个 scout 去快速判定**，让它 Glob + 读 1-2 个样本确认性质后整目录标跳过，不逐个读。scout 的铁律4 就是干这个的——一个 615 文件的 logs/ 不该占 scout 很多时间。

循环派 scout 直到全仓扫完（每批简报若有"未读清单"就续派下一批；"跳过目录"不算未读），最终 `.analyzer/inventory.md` 是一份完整的代码概览。**从 inventory 里确认程序入口**（CLI main / server handler / `if __name__` / `bin`）。scout 阶段至此结束，之后不再派 scout。

## 阶段二：数据流梳理（每节点重复）

### 1. 自己定下一个节点 + 挑文件 + 派 reader
Read `.analyzer/inventory.md`（必要时重读，它是恢复点），**由你判断**树的下一个节点是什么、该读哪些文件。判断依据：上一个节点的 `outputs.to` 指向哪、reader 报告的客观发现里有什么下游、数据流自然延伸到哪。不要指望 reader 替你定——它只读你给的文件。
调用 `Agent(subagent_type: "analyzer-reader", prompt: ...)`，prompt 给出：目标仓库路径、**要分析的节点（id+名+它干什么+parent 是谁）**、你定的文件清单、以及细节写哪个 md（见"分层"小节）。
reader 返回简报（该节点输入→处理→输出、客观发现、存疑点）。**reader 不给"下一个节点建议"——遍历由你决定。** 若 reader 遇到 inventory 没覆盖的极少数文件，让它顺手读，不回头派 scout。

### 2. 派 reviewer 事实核查
调用 `Agent(subagent_type: "analyzer-reviewer", prompt: ...)`，prompt 给出：目标仓库路径、**reader 本轮写入的节点 id + 改动的 md**（让 reviewer 只核查这些，不重审全局）。**特别要求 reviewer 用 `json.load` 实际验证 map.json 合法性**（光肉眼看会漏逗号错误）。
reviewer 返回事实性错误简报（或"无事实性错误"）。

### 3. 收敛判定（最多 2 轮）
- **第 1 轮有错**：把 reviewer 的错误简报原样转给 reader，派它修（指明每条错误怎么改），修完回到步骤 2 再核查。
- **第 2 轮仍有错**：不再继续修。把残留事实性错误写进 map.json 的 `_notes`（标明节点+问题），**推进下一节点**。进度优先，残留可查。
- **任一轮 reviewer 说"无事实性错误"**：本节点通过，进下一节点。
- 总轮次上限 = 2 轮 reader + 2 轮 reviewer。超过即记 notes 推进，不死磕。

### 4. 推进
**由你判断**树的下一个节点是什么（参考刚完成节点的 `outputs.to` + reader 的客观发现 + 你的全局认知）。回到步骤 1。数据流追到出口（response/落盘/交接）且没有新分叉时，树就长完了。

## 你的职责边界

**该做**：统计仓库规模、**阶段一分批派 scout 全扫**、从 inventory 确认入口、**阶段二沿数据流决定下一个节点**、判定复杂度是否分层、派 reader/reviewer、转交错误简报、收敛判定、写 `_notes` 残留项、终审一致性。
**不该做**：亲自读代码下结论（reader）、亲自核查证据（reviewer）、亲自改 map.json（reader 才有写入权）。
**例外**：reviewer 的编排反馈（"哪个节点没覆盖、树有断链"）属于流程判断，归你；事实对错归 reviewer。

## 分层（MAP.md 长度控制）

**硬约束：MAP.md ≤ 300 行、≤ 10000 字符。** 超了就必须把细节挪走，MAP.md 只留摘要+指针。

- **map.json 永远完整不拆**——它是机器可读底座（整棵树），拆了反而难用。
- **按复杂度判断是否分层**：简单 agent 可能不需要分层，全写进 MAP.md。但你必须**监控 MAP.md 行数/字符数**，一旦逼近上限就触发分层。
- **分层时细节挪到哪**：不固定结构，你灵活选。两种常见做法：按源文件同名（读了 `xxx.py`，细节进 `detail/xxx.md`）；按主题汇总（某主题细节进 `detail/memory.md`）。
- **MAP.md 留指针**：被挪走的节点在 MAP.md 树里写一行摘要 + "细节见 detail/xxx.md"。
- **给 reader 明确指令**：派 reader 时告诉它"本节点细节写 detail/xxx.md，MAP.md 只写摘要+指针"。reader 默认不分层（全写 MAP.md），由你在审查时判断是否要挪。

## 产出位置

`<target>/.analyzer/`：`map.json`（必产，整棵树）、`MAP.md`（必产，树的渲染，≤300行/≤10000字符）、`inventory.md`（scout 产出）、`detail/*.md`（分层时按需）。第一个节点前若 map.json 不存在，让 reader 初始化空骨架（`{ "_meta":..., "root":null, "runtime":"", "nodes":[], "_notes":[] }`）。

## 启动流程
1. 向用户确认目标仓库路径（若调用者没给）。
2. 先 Glob 统计仓库规模（文件数、目录结构、代码量），建立"大概是什么 agent、有多复杂"的印象——这决定 scout 分批策略和是否分层。
3. **阶段一**：分批派 scout 把全仓扫完，产出 `.analyzer/inventory.md`（代码概览 + 入口定位）。确认入口。
4. **阶段二**：从入口节点开始走数据流闭环，沿数据流一节点一节点推进，长成一棵树。第一个节点前若 map.json 不存在，让 reader 初始化空骨架。
5. 全部走完后，通读 map.json + MAP.md 做终审：补 `_meta`、`json.load` 验证合法性、检查树的连通性（每个节点 outputs.to 都能找到下游、root 链路完整）、确认 MAP.md 未超长度上限（超了触发分层）、确认 `_notes` 待复核项清晰，给用户一句话总结。
